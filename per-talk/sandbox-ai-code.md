# Deep Analysis: Why, and How You Need to Sandbox AI-Generated Code
**Speaker:** Harshil Agrawal | **Company:** Cloudflare (Senior Developer Advocate) | **Date:** 2026-04-08

## Core Beliefs

- **AI-generated code is untrusted code — full stop.** The speaker's central thesis reframes the entire AI coding paradigm away from intelligence and toward security: "Strip away all the hype. Strip away the AI framing. What we are actually doing is running untrusted code from the internet." He explicitly equates LLM-generated code with a code snippet found "on a random website on the internet" — not as hyperbole but as literal functional equivalence. This is a rejection of the assumption that AI code is *better* than random internet code simply because it was generated by a model.

- **Capability-based security is the only correct model.** The speaker is doctrinaire on this: "Don't enumerate what to block. Enumerate what to allow." He frames the blocklist approach as fundamentally flawed because it requires enumerating every possible attack scenario — a losing game. The allow-list approach ("give them keys to just the three rooms they actually need") is presented not as a preference but as the only intellectually honest security posture. This is a direct challenge to how most AI agent frameworks handle tool permissions today (which often default to broad access with exclusion lists).

- **The industry "forgot to apply what we already know."** The speaker is not proposing novel security research. He's arguing that the AI engineering community is willfully ignoring decades of sandboxing wisdom — browsers, mobile OS permissions, process isolation — in its rush to ship. This is both a critique and a strategic positioning: Cloudflare sells the infrastructure that applies known security patterns to AI workloads.

- **Skepticism about the trajectory of autonomous agents.** While acknowledging that autonomous agents "run multi-step workflows without even asking," the speaker frames this not as progress but as escalation of risk. The faster and more autonomous the agent, the more critical sandboxing becomes. He never says "agents are bad" — but his entire framework implies that autonomy without isolation is reckless.

- **Constraints are features, not limitations.** When discussing V8 isolates, the speaker explicitly reframes their limitations (no file system, no process model, resource limits) as advantages for AI-generated code: "For the use case we are talking about... these constraints are actually features. You want the code to be short-lived, constrained, without side effect." This is a design philosophy — the sandbox should be small enough that the blast radius is zero.

## Pain Points

- **The "over-helpful LLM" problem is underappreciated and insidious.** The speaker spends significant time on this specific threat vector — an LLM that "reads your API keys, your database credentials, and your secrets" not because it's malicious but because it's "trying to help you." He calls it "dangerous precisely because its behavior looks reasonable." This is the threat model that keeps him up at night: not adversarial attacks, but well-intentioned code that accidentally accesses sensitive data. The fact that he places this alongside hallucination and prompt injection as a first-class threat reveals how common this pattern is in real deployments.

- **Secret management in sandboxed environments is done wrong almost universally.** The speaker admits personal failure here: "I did follow this pattern for a while" of passing API keys as environment variables into sandboxes. He calls this out as the single most common mistake he sees — and it's telling that he positions it not as a beginner error but as something he himself had to unlearn. The proxy pattern (worker adds auth headers, secret never enters the sandbox) is presented as the corrective, but the implication is that most teams are still getting this wrong.

- **Indirect prompt injection is the genuinely scary threat.** While the speaker covers hallucination and direct injection, he reserves his strongest language for indirect prompt injection: "This is the one that should genuinely scare you." The scenario — an LLM reads an adversarial document as part of its task, and the document contains hidden instructions — is framed as almost unpreventable at the model level. The only defense is sandboxing at the execution layer. This reveals a deep anxiety: the attack surface isn't the user, it's the entire internet of data the LLM might consume.

- **Container cleanup is a real operational burden.** The speaker is unusually specific about this: "Always use try finally, not try catch." The distinction matters — try-catch handles errors but can silently skip cleanup; try-finally guarantees it. He also recommends maximum sandbox lifetimes (defaulting to 10 minutes on Cloudflare). This level of operational specificity suggests he's seen real production incidents caused by orphaned containers — cost overruns and security surface exposure from idle sandboxes sitting around with cached user data.

- **The isolation vs. capability trade-off has no universal answer.** The speaker spends considerable time walking through two different applications that required two fundamentally different sandboxing approaches. He's transparent about the fact that isolates can't do what containers do and vice versa, and that "in practice, you'll probably use both." This isn't a clean architectural story — it's an honest admission that the tooling landscape is fragmented and use-case-dependent.

## Architecture & Technical Patterns

- **Two-tier sandboxing: V8 Isolates for fast execution, Containers for full environments.** The core architecture is a spectrum. Isolates (powered by Cloudflare's Dynamic Worker Isolates API) provide sub-millisecond sandboxed execution with no file system, no process model, and constrained networking. Containers provide full Linux environments with real file systems, process management, and networking — at the cost of seconds of startup time and higher resource consumption. The decision tree is binary: "Does the code need a file system, processes, or package installs? If yes, container. If no, isolates."

- **Worker RPC bindings as the capability surface.** In the isolate approach, the AI code cannot access anything directly. Instead, the worker passes specific bindings into the isolate — a restricted database interface that only exposes `query`, a logger, etc. The AI code calls these through Worker RPC, which routes back to the controlling worker. This is capability-based security implemented as dependency injection: "The only thing inside are what I put there before I locked it."

- **Three-tier networking control.** The speaker outlines a precise networking spectrum: `null` (fully blocked — recommended default), routed through a proxy service (allow-listed domains, logging, rate limits, auth headers), or fully open (explicitly discouraged for untrusted code). This isn't just a feature list — it's a graduated trust model that maps directly to threat severity.

- **Durable Objects as sandbox orchestrators.** In the container architecture, Cloudflare Durable Objects serve as stateful coordinators that manage the lifecycle of each sandbox. This is a specific pattern: a stateful orchestrator (the Durable Object) managing ephemeral stateless compute (the container VM). The Durable Object handles creation, lifecycle, and teardown — which is critical for the cleanup requirements the speaker emphasizes.

- **Proxy pattern for secrets.** Rather than injecting API keys into the sandbox environment, the sandbox makes requests to the worker's endpoint, and the worker adds authentication headers before forwarding to the external service. The secret lives exclusively in the worker's environment, which the sandbox cannot access. This pattern is presented as mandatory, not optional.

- **Hybrid agent architecture: Isolates as the "fast brain," Containers as the "workbench."** The speaker envisions AI agents that use isolates for rapid tool-calling loops (generating a function, running it in milliseconds, iterating) and containers for heavier tasks (building and deploying applications). This is an emerging pattern for multi-modal agent architectures where different execution tiers handle different phases of agent work.

## Hidden Signals

- **"I did follow this pattern for a while" — the proxy pattern admission.** The speaker's confession that he personally made the mistake of passing API keys as environment variables into sandboxes is not just honesty — it's a signal that this mistake is the industry default. If a senior developer advocate at Cloudflare, whose job is to know these systems, made this mistake, the broader market is almost certainly still making it. This is a quiet call-to-action: the product education gap is massive.

- **"The LLM becomes the attack vector not because it was compromised, because it was used as designed against adversarial input."** This is a quietly radical statement. The speaker is saying the problem isn't a bug in LLMs — it's that LLMs are *functioning correctly* when they process adversarial instructions embedded in data. The fix isn't better models; it's isolation. This positions sandboxing as structurally necessary, not a temporary measure until models improve.

- **The specific mention of multi-tenant isolation as a separate concern.** The speaker lists tenant isolation as one of five things to protect, and later says "never share sandboxes between users" with the intensity of someone who has seen a real data leak. The phrase "once the architecture decision is baked in, it's incredibly hard to undo" suggests he's consulted with teams that made this mistake early and paid for it. Multi-tenancy is a problem most AI agent demos simply ignore.

- **The closing pitch is infrastructure, not AI.** The talk ends not with a vision of better AI but with a checklist of security practices and a QR code to Cloudflare documentation. The speaker is positioning Cloudflare not as an AI company but as the *trust layer* for AI applications. The strategic bet: as AI code generation explodes, the bottleneck becomes safe execution, not model quality — and Cloudflare owns the execution substrate.

- **"Most AI agent tool calling, where the model generates function, runs it, and returns the result — well, isolates."** This is an implicit product roadmap. The speaker is telling you that Cloudflare's V8 isolate infrastructure is specifically designed to be the execution layer for AI agent tool-calling loops. This is a direct competitive shot at other serverless platforms and agent frameworks that run tool calls in-process.

## What's NOT Said

- **No discussion of model-level defenses.** The speaker never mentions prompt guardrails, content filtering, fine-tuning for safety, constitutional AI, or any approach that tries to make the model itself safer. His entire defense strategy is at the execution layer. This is a deliberate framing choice: model-level safety is unreliable, infrastructure-level isolation is deterministic. But the absence is notable — it reveals a worldview that treats models as inherently untrustworthy black boxes that cannot be fixed, only contained.

- **No mention of testing or verification of generated code.** Despite being a talk about AI-generated code quality and safety, there is zero discussion of automated testing, static analysis, formal verification, or code review of what the LLM produces before it runs. The sandbox *is* the verification. This is a striking omission that reveals an assumption: generated code is too variable and too frequent to meaningfully review, so you isolate it instead.

- **No discussion of cost optimization or FinOps.** While the speaker mentions that containers "cost money" and resource limits are important, there's no framework for thinking about the economics of sandboxed AI code at scale. No discussion of cold start costs, sandbox pooling, warm pools, or tiered execution. For a talk at a developer conference in 2026, the absence of cost architecture is notable — it suggests the practice is still early enough that teams are solving for security first and cost later.

- **No competitive analysis.** The speaker never mentions Docker, Firecracker, gVisor, WebAssembly components, or any other sandboxing technology. He never positions Cloudflare's approach against AWS Lambda, Fly.io, Railway, or E2B. This is either strategic (avoid giving airtime to competitors) or revealing (the market is so early that competitive positioning is premature).

- **No mention of the developer experience of being sandboxed.** The talk focuses entirely on the security architect's perspective. There's no discussion of how sandboxing constraints affect the LLM's ability to write useful code, whether models produce worse code when they know they're constrained, or how to communicate sandbox limitations to the model in prompts. The developer/agent experience of operating inside a sandbox is unexplored.

- **No regulatory or compliance framing.** Despite 2026 being a year of active AI regulation (EU AI Act, etc.), the speaker never invokes compliance, audit requirements, or legal liability as motivation for sandboxing. The entire argument is technical and practical. This is either a missed opportunity or a deliberate choice to keep the message universally applicable regardless of jurisdiction.

## Speaker's Worldview

Harshil Agrawal views AI code generation through the lens of an infrastructure security engineer, not an AI researcher. He fundamentally does not trust the model — not because he's hostile to AI, but because he believes trust is the wrong framework entirely. In his worldview, the solution to unsafe AI code is not better models or better prompts; it's deterministic, capability-based isolation borrowed from decades of operating system and browser security. He is pragmatic, anti-hype, and deeply concerned that the industry's excitement about shipping speed is blinding it to risks that are well-understood and solvable — if anyone bothered to apply what they already know.
